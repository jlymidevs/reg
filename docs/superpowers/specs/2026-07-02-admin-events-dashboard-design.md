# Admin Events Dashboard — Design Spec

**Date:** 2026-07-02
**Status:** Approved by user, ready for implementation plan
**Repo:** New, separate repo (name TBD at plan/scaffold time, e.g. `jlycc-admin`) — NOT part of the `jlycc-platform` monorepo.

## Goal

Staff currently create events by having Claude run raw SQL inserts against Supabase, and collect event-specific registrant info (e.g. "which ministry are you in?") via ad-hoc Google Forms. This app replaces both: staff create/edit/publish events themselves, and toggle a couple of common per-event questions (ministry, network) without needing a new Google Form each time.

This is phase 1 of a larger roadmap (see `docs/superpowers/plans/2026-07-02-public-registration-flow.md`'s follow-on work): payments and full admin registrant CSV/export tooling are separate, later specs. This spec covers **event CRUD + registrant list/manual-confirm/CSV-export + the ministry/network registrant fields**.

## Non-goals (explicitly deferred)

- Payments / paid events (separate spec)
- Full Google-Forms-style custom question builder (user chose fixed extra fields instead — see below)
- Waitlist / capacity auto-management (separate spec)
- Audit log table (separate spec, though every admin mutation in this app should already be structured so audit logging can be added later without a rewrite — write mutations as discrete named Server Actions, not ad-hoc inline queries)

## Architecture

- **New repo**, Next.js 15 App Router, Tailwind 4, shadcn/ui for admin UI patterns (data tables, dialogs, forms, file upload).
- **Same Supabase project** as `jlycc-platform` (`gadjquxavyxsftnwurfo`) — same `members`, `roles`, `user_roles`, `events`, `event_registrations`, `ministries`, `networks` tables.
- **Auth**: Google OAuth via Supabase Auth, mirroring `apps/registration/app/login/page.tsx` in the existing repo. Middleware gates all routes except `/login`/`/auth/callback` behind `is_admin()` or `has_role('pcm_staff')` (reuse the same RLS helper functions already defined in `010_rls_policies.sql`).
- **Code reuse strategy**: duplicate `packages/supabase/src/{client,admin,middleware}.ts` (~100 lines total) into the new repo's own `lib/supabase/` — no shared npm package, no monorepo merge. These files are stable; divergence risk is low and doesn't justify publish/versioning overhead.
- **Writes**: Server Actions using a service-role admin client (same `createAdminClient()` pattern as `apps/registration/lib/registration/actions.ts`) — no direct anon/authenticated RLS writes for admin mutations, consistent with the existing "no anon RLS writes" architecture.
- **Image storage**: new Supabase Storage bucket `event-images`, public read policy, write restricted to `is_admin()`/`pcm_staff` via storage RLS. Client uploads directly to a signed upload URL (generated by a Server Action) rather than routing file bytes through the Server Action body.
- **Deploy**: separate Vercel project under `jlymicentral-2498s-projects` (JLY family scope), own domain.

## Database changes (migration lives in `jlycc-platform`, applied there — the new admin repo has no migrations of its own)

```sql
-- 015_admin_event_fields.sql
-- Per-event registrant-question toggles, image, deadline. Additive, idempotent.

alter table public.events
  add column if not exists collect_ministry boolean not null default false,
  add column if not exists collect_network boolean not null default false,
  add column if not exists image_url text,
  add column if not exists registration_deadline timestamptz;

alter table public.event_registrations
  add column if not exists ministry_id uuid references public.ministries(id),
  add column if not exists network_id uuid references public.networks(id);
```

- `collect_ministry` / `collect_network`: per-event booleans the admin sets when creating/editing an event. When true, the corresponding dropdown appears on the **public** registration form (`apps/registration`) sourced live from `public.ministries`/`public.networks` — no per-event custom option lists.
- `ministry_id` / `network_id` on `event_registrations`: nullable FKs, populated only when the event's toggle is on.
- `image_url`: public Storage URL set after the admin uploads an image; nullable, event cards/detail render a placeholder when absent.
- `registration_deadline`: nullable timestamp; when set and passed, the public detail page shows "registration closed" instead of the Register button (small follow-on change to `apps/registration/app/events/[id]/page.tsx`, not part of the admin-app build itself but required for this feature to have any effect).

**Note:** the public `apps/registration` app needs a follow-on update (render ministry/network dropdown when toggled, respect `registration_deadline`) — tracked as a second task after the admin app's own migration + backend work, before the feature is fully end-to-end.

## Admin App Pages

| Route | Purpose |
|---|---|
| `/login` | Google OAuth sign-in |
| `/auth/callback` | OAuth callback handler (same shape as existing repo's) |
| `/events` | List all events regardless of published state; status badges (draft/published/past); inline publish/unpublish toggle |
| `/events/new` | Create form: title, description, event_type, venue, starts_at, ends_at, capacity, requires_registration, registration_deadline, image upload, collect_ministry toggle, collect_network toggle |
| `/events/[id]/edit` | Same form pre-filled; includes a "deactivate" action (`is_active = false`) in place of hard delete, to avoid orphaning existing registrations |
| `/events/[id]/registrants` | Table: name, contact (email/mobile), status, ministry/network (if collected for this event), matched-member vs guest indicator; filter by status; CSV export button; per-row action to confirm a `pending_review` guest or cancel a registration |

## Data Flow

1. Admin creates/edits an event → Zod-validated Server Action (schema mirrors the DB constraints, same pattern as `apps/registration/lib/registration/schema.ts`) → service-role insert/update on `events`.
2. Image upload: client requests a signed upload URL from a Server Action → uploads directly to Storage → on success, a second Server Action call sets `events.image_url`.
3. Registrant table reads via a Server Action (or RSC data fetch) using the service-role client, since registrant PII crosses beyond what a plain-authenticated staff RLS policy currently grants (existing `event_registrations_select` policy scopes to `can_access_member(member_id)`, which doesn't cover guest rows) — the admin app's registrant list is therefore server-role-mediated, matching how `apps/registration`'s own actions already work.
4. Manual confirm/cancel: Server Action updates `event_registrations.status`, validates the transition (can't confirm an already-cancelled row, etc.).
5. CSV export: Server Action queries registrants for the event, serializes to CSV, returns as a downloadable response. No export queue/job needed at current volume (dozens–hundreds of registrants per event).

## Error Handling

- Zod validation errors surface inline on the form, same UX pattern as the existing registration form.
- Storage upload failures show a retry-able error state, don't block the rest of the event form from saving (image is optional).
- Status-transition validation (e.g., confirm-on-cancelled) returns a typed error result, not a thrown exception, consistent with `RegisterResult`'s existing `{ ok: false, error }` shape.

## Testing

- Vitest unit tests: event Zod schema (mirrors `schema.test.ts` pattern), registrant status-transition guard logic.
- Manual end-to-end checklist before first real staff use: create an event with an image and both toggles on → confirm it appears correctly on the public site (after the registration-app follow-on ships) → register as a guest with ministry/network selected → confirm the row appears in the admin registrants list with the right ministry/network → manually confirm the `pending_review` row → export CSV and confirm the row is present with correct data.
